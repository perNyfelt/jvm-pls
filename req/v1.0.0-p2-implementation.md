# Phase 2 Implementation Plan

## Goal

Implement Phase 2 from [`req/v1.0.0-roadmap.md`](/home/per/project/jvm-pls/req/v1.0.0-roadmap.md): classpath and external type resolution for JDK and dependency JARs.

The target outcome is that external symbols resolve through the same `CoreQuery` path that source symbols already use, so existing Java and Groovy plugin logic can start resolving JDK and classpath types with minimal special-casing.

## Scope

Phase 2 covers:

- a new `jvmpls-classpath` module
- lightweight classpath scanning
- lazy binary type reading
- a classpath-backed symbol provider
- JDK indexing through `jrt:/`
- core integration so `SymbolIndex` can resolve external symbols

Phase 2 does not need to fully solve:

- build-tool auto-detection and workspace model resolution
- rich structured type modeling
- receiver-type-aware dotted completions

Those belong to later phases.

## Architectural Direction

The core design principle is:

- external symbols must be discoverable through `CoreQuery`
- source symbols remain first-class and eagerly indexed
- classpath symbols are provided lazily on lookup miss

This keeps existing plugin code viable because both plugins already resolve symbols through `CoreQuery` and `SymbolIndex`.

## Workstreams

### 1. Add `jvmpls-classpath`

Create a new Maven module:

- `jvmpls-classpath`

Dependencies:

- `jvmpls-core`
- ClassGraph
- ASM

Primary classes to introduce:

- `ClasspathScanner`
- `BinaryTypeReader`
- `ClasspathSymbolProvider`
- `JdkIndex`

### 2. Extend `SymbolIndex` with provider support

Current `SymbolIndex` only stores source symbols in memory. Phase 2 should extend it with lazy provider registration.

Add:

- `registerProvider(SymbolProvider provider)`

Provider lookup behavior:

- `findByFqn` checks in-memory source symbols first, then providers
- `findBySimpleName` merges in-memory and provider results
- `allInPackage` merges in-memory and provider results

Important constraints:

- source symbols win over external symbols on conflicts
- provider results should be cached
- provider lookup should not trigger repeated full scans

### 3. Define a `SymbolProvider` interface

Add a provider contract in `jvmpls-core`, shaped to existing query usage:

```java
public interface SymbolProvider {
    Optional<SymbolInfo> findByFqn(String fqn);
    List<SymbolInfo> findBySimpleName(String simpleName);
    List<SymbolInfo> allInPackage(String pkgFqn);
}
```

This should stay query-oriented for Phase 2. Do not mix in Phase 4 structured type APIs yet.

### 4. Implement `ClasspathScanner`

`ClasspathScanner` should use ClassGraph to scan:

- classpath directories
- dependency JARs
- optionally the JDK module image via `jrt:/`

Its output should be lightweight metadata, not full symbol hydration.

Recommended metadata shape:

- FQN
- package name
- symbol kind
- classpath origin
- superclass name
- interface names
- modifiers

This metadata is enough for package/type lookup and later lazy hydration.

### 5. Implement `BinaryTypeReader`

`BinaryTypeReader` should use ASM to read `.class` files on demand.

Responsibilities:

- class kind
- method signatures
- field types
- annotation values
- generic signature strings
- parameter names when available

This should be lazy. It should only read a class when the provider actually needs richer `SymbolInfo` materialization.

Phase 2 should keep this pragmatic:

- raw signature strings are acceptable
- no need to introduce full `JvmType` conversion yet

### 6. Implement `ClasspathSymbolProvider`

`ClasspathSymbolProvider` should adapt scanner output and binary reads into `SymbolInfo`.

Responsibilities:

- satisfy `findByFqn`
- satisfy `findBySimpleName`
- satisfy `allInPackage`
- cache materialized external symbols

Implementation strategy:

- package and type symbols first
- methods and fields only if needed to support current lookups cleanly

The first working milestone should be external package and type resolution, since that unblocks imports, basic completions, hovers, and definitions.

### 7. Implement `JdkIndex`

`JdkIndex` should scan Java 9+ `jrt:/`.

Resolution rules:

- prefer the target JDK configured for the workspace
- if none is provided, fall back to the running JDK

`JdkIndex` should feed the same provider path as normal classpath symbols. Do not build a separate JDK-only resolution mechanism.

### 8. Integrate with core startup

Update core startup so a classpath provider can be registered during engine initialization.

Likely touchpoints:

- `SymbolIndex`
- `CoreEngine`
- `CoreServer`

Expected changes:

- `CoreEngine` accepts classpath entries and target-JDK configuration
- startup builds the scanner/provider and registers it with `SymbolIndex`
- external type resolution becomes available immediately after configuration

### 9. Provide manual configuration first

Do not block Phase 2 on Phase 3 build-system plugin work.

Initial configuration path should support:

- explicit classpath entries
- explicit target JDK path

This can be consumed by:

- embedded mode first
- server initialization later

Automatic Maven/Gradle resolution is a Phase 3 concern.

### 10. Test strategy

#### Unit tests

Add focused tests for:

- `SymbolIndex` provider fallback
- provider result caching
- duplicate suppression between source and external symbols
- package and simple-name lookups

#### `jvmpls-classpath` tests

Add tests for:

- scanning a small fixture JAR
- reading binary type metadata from `.class`
- JDK scanning via `jrt:/`

#### Integration tests

Add end-to-end tests proving:

- Java resolves `java.util.List`
- Groovy resolves `java.util.Map`
- completions include external package/type symbols
- definitions on external symbols return a usable external location representation

## Recommended Sequence

Implement in this order:

1. Add `SymbolProvider`
2. Extend `SymbolIndex` with provider registration and fallback
3. Create `jvmpls-classpath`
4. Implement `ClasspathScanner`
5. Implement `ClasspathSymbolProvider`
6. Implement `JdkIndex`
7. Implement `BinaryTypeReader`
8. Add `CoreEngine` and `CoreServer` configuration hooks
9. Add Java and Groovy integration tests

This sequence gets a usable external-type vertical slice early, then deepens it.

## Risks

### Scope creep into Phase 4

The biggest risk is dragging Phase 4 type-system work into Phase 2. Avoid that. Phase 2 only needs enough binary metadata to resolve external package and type symbols reliably.

### Performance regressions

Classpath scanning can become expensive if every lookup re-reads classes. Keep the scan lightweight and cache materialized symbols.

### Incorrect abstraction split

If `ClasspathSymbolProvider` bypasses `CoreQuery`, plugin behavior will diverge between source and binary symbols. Keep all external resolution behind `SymbolIndex` and provider registration.

### Dependency on build tool resolution

Phase 2 should ship with manual classpath configuration first. Waiting for Maven/Gradle integration would unnecessarily block the phase.

## Definition of Done

Phase 2 is done when:

- `SymbolIndex` supports lazy external symbol providers
- JDK types such as `String`, `List`, and `Map` resolve through normal plugin lookups
- dependency JAR types resolve through the same mechanism
- Java and Groovy completions can surface external types
- definitions for external types succeed without crashing
- all of the above are covered by unit and integration tests
