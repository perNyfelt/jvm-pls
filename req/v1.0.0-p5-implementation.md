# Phase 5 Implementation Plan

## Goal

Implement Phase 5 from [`req/v1.0.0-roadmap.md`](/home/per/project/jvm-pls/req/v1.0.0-roadmap.md): Groovy dynamic feature support.

The target outcome is that Groovy code no longer stops at the static AST boundary. The server should understand common compile-time AST transforms and the highest-value runtime metaprogramming patterns well enough to:

- surface synthetic methods, fields, constructors, and builder APIs in completion and definition
- reduce false "unknown method/property" behavior in dynamic Groovy code via a narrow Groovy semantic pass
- preserve Phase 4 typed symbol behavior when static information exists
- degrade gracefully, with explicit confidence, when only heuristic dynamic information is available

## Scope

Phase 5 should cover:

- a new `se.alipsa.jvmpls.groovy.transforms` package
- a new `se.alipsa.jvmpls.groovy.dynamic` package
- additive constructor-capable reporting/query support in the core model
- compile-time AST transform analyzers for the built-in Groovy transforms listed in the roadmap
- runtime metaprogramming heuristics for `metaClass`, categories, mixins, `methodMissing`, and `propertyMissing`
- synthetic symbol reporting through the existing Groovy indexing path
- a plugin-local scoped synthetic-member overlay for lexical constructs such as `use(Category) { ... }`
- additive core-model support for synthetic provenance and confidence
- a shared Groovy member-resolution helper used by completion, definition, and the narrow semantic diagnostic pass
- confidence-aware ranking for heuristic-only dynamic symbols
- a narrow Groovy semantic diagnostic pass for undefined member/property checks in contexts Phase 5 understands

Phase 5 should not try to finish:

- framework-specific Groovy behavior for Grails, Gradle DSLs, or Spock beyond extension hooks
- full runtime execution or sandboxed evaluation of Groovy code
- rename/refactor support across dynamic symbols
- full semantic diagnostics from Phase 6

Known limitation for Phase 5:

- dynamic behavior inferred from heuristics remains best-effort; if code computes member names or metaprogramming targets indirectly, the server should prefer graceful fallback over speculative precision

## Architectural Direction

Phase 5 should extend the existing Groovy plugin rather than creating a separate dynamic engine.

That means:

- `GroovyPlugin.index()` remains the single place where Groovy source is parsed and symbol data is produced
- compile-time transforms and global runtime heuristics should emit synthetic `SymbolInfo` through the same reporting path used for ordinary declarations
- lexical-scope dynamic additions such as category methods should live in a Groovy-plugin-local scoped overlay keyed by file and source range, then be merged into completion/definition results at query time
- dynamic results should integrate with the Phase 4 type system instead of bypassing it
- uncertainty should be modeled explicitly, not hidden

Four design rules should be explicit:

- compile-time transforms should be handled structurally and deterministically from annotations plus AST context
- runtime metaprogramming should be heuristic and confidence-scored
- static symbols always outrank synthetic symbols, and high-confidence synthetic symbols outrank low-confidence heuristic ones
- `DynamicType` remains the fallback marker when the plugin knows a context is dynamic but cannot resolve a precise static type

Two additional implementation rules should also be explicit:

- transformed constructors are first-class outcomes of Phase 5, so constructor symbols need an explicit reporting/query path rather than being smuggled through method-only APIs
- scoped dynamic visibility must be resolved through one shared Groovy member-resolution path so completion, definition, and the Groovy semantic diagnostic pass cannot drift

## Core Design Decisions

### 1. Synthetic symbols should stay additive

Phase 5 should not mutate or replace existing source symbols. Transform and dynamic analyzers should add synthetic members on top of the indexed source model.

### 2. Confidence should be attached to inferred dynamic members

Heuristic symbols need metadata that distinguishes:

- deterministic synthetic members from compile-time transforms
- high-confidence runtime additions such as direct `metaClass.foo = { ... }`
- low-confidence dynamic allowances such as `methodMissing`

This metadata should be available to completion ranking and the Groovy semantic diagnostic pass, so Phase 5 should make additive core model changes mandatory rather than keeping provenance/confidence in a Groovy-only side channel.

### 3. Scope matters for dynamic additions

Not all synthetic members are global:

- category methods only apply inside a `use(Category) { ... }` scope
- mixins may apply per class or per file context
- `metaClass` additions may target a specific type and should only affect that type

The plan should preserve scope boundaries rather than stuffing all inferred members into one global bucket.

For Phase 5 that means two storage modes:

- globally queryable synthetic members go through `SymbolReporter` and the ordinary index
- lexical-only synthetic members, especially category methods, go into a Groovy-plugin-local scoped overlay consulted by completion and definition using the current file and cursor position

### 4. Do not execute user code

Phase 5 should remain source-analysis-only. No Groovy script execution, transform execution, or classloading of workspace code should be introduced.

### 5. `@CompileStatic` and `@TypeChecked` should be recognized explicitly

Phase 5 should not leave stricter static-mode behavior implicit.

Recommended rule:

- these annotations should be recognized directly by Groovy indexing/semantic analysis as mode flags on classes and methods
- they do not need full transform analyzers because their main effect in Phase 5 is to tighten completion/definition/diagnostic behavior rather than synthesize members

## Workstreams

### 1. Add core synthetic-symbol provenance and confidence support

Before implementing analyzers, add additive metadata fields in `jvmpls-core` so the rest of the pipeline can actually see synthetic provenance/confidence.

Recommended additions:

- `SyntheticOrigin` enum or equivalent
- `InferenceConfidence` enum with at least `DETERMINISTIC`, `HIGH`, `LOW`
- additive provenance/confidence fields on `SymbolInfo`
- additive provenance/confidence fields on `CompletionItem`
- typed `SymbolReporter` overloads that can report synthetic provenance/confidence without string hacks

This should remain backward-compatible, but it should be a real core-model change, not a Groovy-only side channel.

### 2. Add transform analyzer SPI

Create `se.alipsa.jvmpls.groovy.transforms.TransformAnalyzer`.

Recommended shape:

- input: Groovy class node, annotation node, indexing context
- output: synthetic members and optional diagnostics/hints

Also add `TransformRegistry` that maps annotation FQNs to analyzers and is easy to extend later for framework plugins.

This should be registry-driven, not a large `switch` in `GroovyPlugin`.

### 3. Add constructor-capable reporting and query support

Before constructor-generating transforms can be implemented cleanly, Phase 5 needs an explicit constructor representation in the core model.

Recommended additions:

- a `CONSTRUCTOR` `SymbolInfo.Kind`, or an equivalent explicit constructor representation
- additive `SymbolReporter` support for constructors
- constructor-aware query support where Groovy completion/definition can enumerate constructors without pretending they are ordinary methods
- alignment with the existing typed `MethodSignature` model from Phase 4

This should be an explicit core change, not an analyzer-local convention around `<init>` or method-name strings.

Classpath/JDK binary constructor support is intentionally not a Phase 5 goal by itself. Phase 5 should stop at introducing the constructor-capable core model plus Groovy/source-side constructor reporting.

Recommended explicit deferral:

- later phases that deepen external symbol UX, especially Phase 6 hover/signature/code-action work, should connect the new constructor model into classpath/JDK symbols
- Phase 5 should therefore design the constructor representation so `jvmpls-classpath` can adopt it later without another model rewrite
- do not imply that classpath/JDK constructor support will arrive automatically just because the core model exists

### 4. Implement core compile-time transform analyzers

Start with the highest-value built-ins from the roadmap:

- `@Builder`
- `@Canonical`
- `@TupleConstructor`
- `@Immutable`
- `@Delegate`
- `@Slf4j` / `@Log` / `@Log4j` / `@Log4j2` / `@Commons`

Then add the next tier:

- `@Sortable`
- `@AutoClone`
- `@AutoExternalize`
- `@Memoized`

Recommended behavior:

- `@Builder` should produce synthetic builder API members and the static `builder()` entry point
- constructor-generating transforms should synthesize constructor signatures with Phase 4 `MethodSignature`
- logging transforms should synthesize the `log` field with the correct logger type when known
- `@Delegate` should copy or project delegate members into the owning class with deterministic origin metadata

### 5. Integrate transform analysis into Groovy indexing

Update `GroovyPlugin.index()` so transform analyzers run after the base AST walk has established the ordinary declarations of each class.

That integration should:

- inspect class annotations
- dispatch matching analyzers from `TransformRegistry`
- report synthetic members through `SymbolReporter`
- ensure transform-produced symbols coexist cleanly with declared members

This sequencing matters: transforms need the underlying class shape first, and later dynamic heuristics should be able to see transform-added members too.

### 6. Add dynamic heuristic analyzer SPI, scoped overlay model, and shared resolution helper

Create `se.alipsa.jvmpls.groovy.dynamic` with a small set of analyzers:

- `MetaClassAnalyzer`
- `MethodMissingAnalyzer`
- `PropertyMissingAnalyzer`
- `CategoryAnalyzer`
- `MixinAnalyzer`

Recommended approach:

- each analyzer scans relevant AST patterns during Groovy indexing
- each emits synthetic members plus confidence metadata
- each can also mark types or scopes as "dynamically extensible" when exact members are not knowable

This workstream should also add a plugin-local scoped-member structure for lexical dynamic constructs. Recommended shape:

- file URI + source range + target receiver type
- synthetic members visible only inside that range
- merge logic in Groovy completion/definition paths before falling back to `core.membersOf(...)`

Phase 5 should not try to retrofit lexical category scope into the current global `membersOf(String ownerFqn)` API.

This workstream should also add one shared Groovy member-resolution helper that merges:

- declared/static members from the core index
- globally reported synthetic members
- plugin-local scoped overlay members
- confidence/provenance metadata
- current static-mode flags such as `@CompileStatic` / `@TypeChecked`

That helper should be the single path used by:

- dotted completion
- definition on Groovy members
- the narrow undefined-member semantic diagnostic pass

### 7. Implement `metaClass` support first

This should be the first runtime heuristic because it has the clearest source pattern and highest editor value.

Recognize cases like:

- `Foo.metaClass.bar = { ... }`
- `Foo.metaClass.getBar = { ... }`
- `Foo.metaClass.static.baz = { ... }`

Phase 5 should infer:

- target type
- synthetic member/property name
- whether it is instance or static
- approximate parameter/return types when the closure signature provides enough information

This should be treated as `HIGH` confidence when the target and member name are explicit.

### 8. Implement `methodMissing` and `propertyMissing`

These analyzers should not invent arbitrary concrete methods. Instead they should:

- mark the class as dynamically extensible
- influence completion and diagnostics fallback behavior
- optionally surface low-confidence placeholder entries only when that improves UX

The main value here is to avoid incorrect hard failures in obviously dynamic Groovy classes.

### 9. Implement category support

Recognize `use(CategoryType) { ... }` blocks and temporarily expose category methods on matching receiver types inside the block scope.

This is the main scope-sensitive dynamic feature in Phase 5, so the implementation should explicitly model:

- block entry/exit scope
- target receiver matching
- member visibility only within the `use` region

Implementation rule:

- category methods should not be reported into the global symbol index
- they should live only in the plugin-local scoped overlay and be consulted through the shared Groovy member-resolution helper with file/position context

### 10. Implement mixin support

Handle:

- `@Mixin(Foo)`
- `Target.mixin(Foo)`

The analyzer should add synthetic members from the mixed-in type to the target type with clear synthetic provenance.

Where exact semantics are ambiguous, prefer conservative exposure of callable members over trying to perfectly reproduce runtime dispatch order.

### 11. Extend completion ranking for synthetic and heuristic members

Update Groovy completion logic so ordering becomes:

1. declared/static known members
2. deterministic synthetic members from transforms
3. high-confidence dynamic heuristic members
4. low-confidence dynamic fallback members

This avoids polluting normal completion with speculative results while still making dynamic Groovy usable.

### 12. Add a narrow Groovy semantic diagnostic pass and dynamic-context relaxation

The current Groovy plugin does not yet have a semantic undefined-member diagnostic pass, so Phase 5 needs a small prerequisite here rather than assuming one already exists.

Recommended scope:

- dotted property access
- dotted method calls
- explicit member references in contexts already analyzed for completion/definition

Implementation rule:

- this pass should resolve members only through the shared Groovy member-resolution helper added earlier
- it should not reimplement scoped overlay lookup, provenance handling, or static-mode filtering independently

This pass should then add Groovy-specific relaxation for:

- `methodMissing`
- `propertyMissing`
- clear dynamic `metaClass`/mixin/category scopes

Recommended behavior:

- in static contexts, keep the stricter Phase 4 behavior
- in known dynamic contexts, downgrade "undefined method/property" from hard failure to softer warning or suppress it when a heuristic synthetic symbol exists
- if `@CompileStatic` or `@TypeChecked` is present, use the explicit static-mode flags recorded during Groovy indexing and keep semantics stricter

### 13. Add extension hooks for future framework analyzers

Without implementing Grails/Gradle/Spock now, leave clean extension points:

- transform registry extensibility
- dynamic analyzer registry extensibility
- package-level conventions for registering framework-specific logic

Phase 5 should avoid hard-coding assumptions that make later framework analyzers awkward to add.

## Recommended Sequence

1. core synthetic provenance/confidence model
2. constructor-capable reporting and query support
3. transform analyzer SPI and registry
4. first transform analyzers: `@Builder`, constructor transforms, log transforms, `@Delegate`
5. integrate transform analysis into `GroovyPlugin.index()`
6. dynamic heuristic SPI, scoped overlay model, and shared resolution helper
7. `MetaClassAnalyzer`
8. `MethodMissingAnalyzer` and `PropertyMissingAnalyzer`
9. `CategoryAnalyzer`
10. `MixinAnalyzer`
11. completion ranking updates
12. narrow Groovy semantic diagnostic pass plus dynamic-context relaxation
13. extension hooks and cleanup

## Test Strategy

Phase 5 needs more than integration-only coverage. Add focused unit tests for each analyzer category plus a few end-to-end Groovy completions/definitions.

Recommended test categories:

- transform analyzer unit tests:
  - `@Builder`
  - `@Canonical` / `@TupleConstructor`
  - `@Immutable`
  - `@Delegate`
  - log transforms
- dynamic analyzer unit tests:
  - direct `metaClass` method/property additions
  - `methodMissing`
  - `propertyMissing`
  - `use(Category)` scoped methods
  - mixin additions
- core-model tests:
  - provenance/confidence propagation through `SymbolInfo`, `SymbolReporter`, and `CompletionItem`
  - constructor reporting/query behavior for transformed constructors
  - shared Groovy member-resolution behavior for global plus scoped synthetic members
- Groovy plugin integration tests:
  - transformed members appear in completion
  - synthetic constructors resolve in completion/definition paths
  - dynamic members rank below static members
  - category methods disappear outside the `use` block
  - dynamic contexts soften the narrow undefined-member diagnostic pass correctly
  - `@CompileStatic` / `@TypeChecked` contexts stay stricter than plain dynamic Groovy

## Main Risks

- overexposing speculative dynamic members and drowning out real static results
- leaking scoped dynamic members globally
- implementing built-in transforms with behavior too tied to string heuristics instead of AST structure
- letting Groovy dynamic support fork away from the Phase 4 typed-symbol model
- leaving constructor support half-migrated if the new model is added in Phase 5 but not explicitly adopted by classpath/JDK work in the next feature phase
- trying to perfectly emulate Groovy runtime semantics instead of shipping strong, conservative heuristics

## Success Criteria

Phase 5 should be considered complete when:

- `@Builder` and constructor-generating transforms appear in completions and definitions
- log transforms expose a usable `log` field
- `@Delegate` members are visible on the owning type
- direct `metaClass` additions appear on target receivers
- `methodMissing` / `propertyMissing` classes no longer generate obviously wrong results in completion/definition, and the narrow undefined-member diagnostic pass no longer hard-fails there
- category methods are available only inside their `use` scopes
- mixin-provided methods are visible on the mixed target type
