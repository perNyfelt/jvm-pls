# Phase 4 Implementation Plan

## Goal

Implement Phase 4 from [`req/v1.0.0-roadmap.md`](/home/per/project/jvm-pls/req/v1.0.0-roadmap.md): a structured JVM type system and enhanced symbol resolution.

The target outcome is that the server stops reasoning about types as loosely formatted strings and instead carries structured type information end to end. That should unlock:

- receiver-type-aware dotted completions
- overload-aware method and constructor resolution
- generic type propagation through source and binary symbols
- a shared type foundation for hover, signature help, references, diagnostics, and code actions in later phases

## Scope

Phase 4 should cover:

- a new `se.alipsa.jvmpls.core.types` package in `jvmpls-core`
- the `JvmType` hierarchy and structured signatures
- source-context type resolution for Java and Groovy indexing/query paths
- binary member capture plus signature mapping from ASM into structured types
- type-relationship queries for assignability and hierarchy traversal
- lazy member lookup in the core query/provider path
- dotted completion receiver resolution
- constructor and overload selection support in the core query path

Phase 4 should not try to finish:

- full semantic diagnostics from Phase 6
- rename or reference indexing
- every Groovy dynamic feature from Phase 5
- full framework-specific typing rules for Gradle DSL, Grails, Spock, etc.

Known limitation for Phase 4:

- multi-module scoping remains workspace-global from Phase 3, so type lookup can still see broader classpaths than ideal

## Architectural Direction

Phase 4 should centralize type knowledge in `jvmpls-core`, not in the language plugins.

That means:

- language plugins should produce enough syntax/semantic context to resolve types
- `jvmpls-core` should own the shared type model, type relations, and method/constructor selection logic
- `jvmpls-classpath` should map binary signatures into the same `JvmType` model used by source symbols
- later LSP features should consume the same structured types rather than re-parsing strings

Two design constraints should be explicit:

- `SymbolInfo` already exists and is used widely, so the transition must be additive first, not a flag day rewrite
- Groovy needs an explicit `DynamicType` that is distinct from `Object`, otherwise the completion engine will become too pessimistic in dynamic contexts

Three API decisions should also be explicit up front:

- member lookup must be added to the core query/provider layer before dotted completion and overload resolution can be implemented cleanly
- Phase 4 should not change the existing method FQN key format; overload handling should move to member-query APIs rather than key rewrites
- `SymbolReporter` and `CompletionItem` both need additive typed extensions so typed metadata can flow from plugins into the core and then out to the LSP layer

## Workstreams

### 1. Add the core type model

Create `se.alipsa.jvmpls.core.types` in `jvmpls-core`.

Add:

- `JvmType` as a sealed interface
- `PrimitiveType`
- `ClassType`
- `ArrayType`
- `TypeVariable`
- `WildcardType`
- `VoidType`
- `DynamicType`

Design requirements:

- all implementations should be immutable
- `ClassType` should carry FQN plus type arguments
- `TypeVariable` should preserve the declared name and optional bounds
- `DynamicType` should be a first-class marker, not a string alias

### 2. Add structured signatures

Introduce `MethodSignature` in `jvmpls-core`.

It should capture:

- parameter types
- return type
- type parameters
- throws types
- modifiers
- optional parameter names when known

This should become the canonical representation for callable symbols. String signatures can remain for compatibility during transition, but new logic should prefer `MethodSignature`.

### 3. Extend `SymbolInfo` additively

Update `SymbolInfo` to carry structured type information without breaking existing code immediately.

Recommended additions:

- `JvmType resolvedType()` for fields, locals, parameters, variables, and property-like members
- optional `MethodSignature methodSignature()` for methods and constructors

Phase 4 should preserve legacy string fields where needed, but new code paths should stop deriving behavior from raw string signatures.

### 4. Extend reporting and completion data additively

Typed metadata needs a way to move through the existing indexing and completion pipeline.

Recommended changes:

- add typed overloads to `SymbolReporter` for fields, methods, and constructors
- preserve the legacy string-based `reportMethod` and `reportField` entry points during transition
- extend `CompletionItem` with optional type/display metadata for field types, return types, and parameter display
- update `LspTypeConverter` later in the phase to surface that data in completion detail/documentation

Phase 4 should keep the old reporting path working until both language plugins and classpath symbols are migrated.

### 5. Add member lookup to the core query layer

`CoreQuery` and the provider path need member-aware lookup before dotted completion can exist.

Recommended additions:

- owner-scoped member lookup such as `membersOf(String ownerFqn)`
- optional filtered lookups for methods, fields, and constructors
- lazy provider-backed member lookup for external/classpath symbols

This should be implemented without changing existing FQN keys. Existing exact-key lookup can remain for compatibility, but new overload/member logic should stop depending on key parsing.

### 6. Extend `BinaryTypeReader` to capture members

Before binary signatures can be mapped to `JvmType`, the ASM visitor needs to actually collect method and field data.

This workstream should add:

- `visitField` handling
- `visitMethod` handling
- capture of descriptors and generic signatures
- capture of modifiers for fields and methods
- capture of thrown types and parameter names where available

This is also where the current modifiers gap should be closed for external members, so visibility filtering is possible later.

### 7. Add lazy external member materialization

[`ClasspathSymbolProvider`](/home/per/project/jvm-pls/jvmpls-classpath/src/main/java/se/alipsa/jvmpls/classpath/ClasspathSymbolProvider.java) currently materializes only top-level types.

Phase 4 should add lazy member materialization for an owning type, driven by the new member-query API rather than eager global indexing.

That means:

- member `SymbolInfo` instances for methods, fields, and constructors are produced on demand
- classpath members use the same `SymbolInfo` and future typed metadata model as source symbols
- dotted completion and overload selection can enumerate members of `java.util.List`, `Map`, etc.

### 8. Map binary members and signatures into `JvmType`

After member capture exists, extend [`BinaryTypeReader`](/home/per/project/jvm-pls/jvmpls-classpath/src/main/java/se/alipsa/jvmpls/classpath/BinaryTypeReader.java) so binary symbols stop exposing only raw generic signature strings.

Add:

- mapping from ASM type descriptors to `JvmType`
- mapping from generic signatures to `ClassType`, `TypeVariable`, and `WildcardType`
- field type materialization as `JvmType`
- method and constructor materialization as `MethodSignature`

This is the bridge that makes external JDK/JAR members behave like source members in later queries.

### 9. Add type reference resolution

Implement `TypeResolver` in `jvmpls-core`.

Responsibilities:

- resolve primitive names
- resolve arrays
- resolve simple names through imports and package scope
- resolve fully qualified names
- resolve declared generic arguments
- resolve source and binary types through `SymbolIndex`

This should be context-aware. It needs enough source context to answer questions like:

- what does `List<String>` mean here
- what does `Foo` refer to in this file
- what does `T[]` mean inside this generic class or method

### 10. Add type relations

Implement `TypeRelations` in `jvmpls-core`.

Required queries:

- `isAssignableTo`
- `isSubtypeOf`
- `commonSupertype`

It should use:

- the source symbol hierarchy already in `SymbolIndex`
- external type hierarchy data from `jvmpls-classpath`
- special handling for primitives, boxing, arrays, wildcards, and `DynamicType`

Recommended Phase 4 behavior:

- `DynamicType` should be treated as “plausibly assignable” for completion/ranking paths
- diagnostics-grade strictness can come later
- it is acceptable to implement source-symbol relations first, but external hierarchy support must land before receiver-type-aware completion is considered done

### 11. Update Java plugin indexing

Update `jvmpls-java` so indexed symbols populate structured type information.

Targets:

- field declarations
- method return types
- method parameters
- constructor symbols
- local variables where needed for completion receiver inference

The Java path should be the first strict implementation because the syntax is the least ambiguous.

This workstream should also add modifier propagation for source methods and fields, since the current reporting path drops that data.

### 12. Update Groovy plugin indexing

Update `jvmpls-groovy` to populate structured type information where types are statically known and use `DynamicType` where they are not.

Targets:

- explicit type annotations
- inferred declaration types when the AST provides them reliably
- method parameters and return types
- dynamic/property access contexts where inference is intentionally uncertain

The key rule is:

- unknown Groovy should become `DynamicType`
- not `java.lang.Object`

That preserves useful completion behavior for dynamic code.

### 13. Add receiver-type-aware dotted completion

Refactor the completion pipeline so `foo.bar` first resolves the type of `foo`, then offers members of that type.

This requires:

- local variable and parameter type lookup
- field/property type lookup
- method-call return type propagation where available
- member enumeration over source and binary types
- inherited member lookup through `TypeRelations`

Phase 4 should prefer working dotted completions for common expressions over perfect inference in every edge case.

### 14. Add overload and constructor selection

Implement a shared callable resolution utility in `jvmpls-core`.

Responsibilities:

- rank overload candidates by parameter compatibility
- resolve constructors for `new Foo(...)`
- propagate generic substitutions where practical
- handle binary and source callables uniformly

This does not need to expose a full public “compiler-like” API yet; it just needs to support completion, definition, and later signature help.

### 15. Refactor the remaining core query paths to use types

After member lookup exists, review the remaining `CoreQuery`, `CoreFacade`, and plugin query paths so type-bearing symbols can flow through them without repeated string parsing.

Recommended direction:

- keep the public query surface stable where possible
- introduce internal helpers for typed member lookup and callable resolution
- ensure new type-aware behavior is available to both Java and Groovy plugins through shared core utilities

### 16. Maintain backward compatibility during transition

Phase 4 should be implemented in two layers:

1. additive typed fields and utilities
2. migration of query/completion logic to the typed path

Do not remove legacy string signature fields until:

- Java and Groovy indexers are both migrated
- classpath symbols expose structured types
- the tests prove parity for current features

### 17. Test strategy

#### Unit tests in `jvmpls-core`

Add focused tests for:

- `JvmType` equality and rendering
- `TypeResolver` import/package/generic resolution
- member lookup and owner-scoped query behavior
- `TypeRelations` assignability and common supertype
- overload ranking and constructor selection

#### Classpath tests in `jvmpls-classpath`

Add tests for:

- binary member capture from ASM
- descriptor to `JvmType` mapping
- generic signature parsing
- binary methods and fields exposing `MethodSignature` and `JvmType`

#### Java plugin tests

Add tests proving:

- dotted completion on typed receivers
- inherited member completion
- constructor resolution
- overload ranking for simple argument lists

#### Groovy plugin tests

Add tests proving:

- explicit types become structured `JvmType`
- dynamic contexts use `DynamicType`
- dotted completions remain permissive in dynamic cases

#### Server/integration tests

Add end-to-end tests proving:

- JDK/JAR methods are offered on typed receivers
- definitions can navigate through typed member resolution
- completion detail/type display is populated from structured type metadata
- generics do not regress existing completions/definitions

## Recommended Sequence

1. add `JvmType` and `MethodSignature`
2. extend `SymbolInfo`, `SymbolReporter`, and `CompletionItem` additively
3. add member lookup to `CoreQuery` and the provider layer
4. extend `BinaryTypeReader` to capture methods, fields, modifiers, and descriptors
5. add lazy external member materialization in `ClasspathSymbolProvider`
6. map binary members and signatures into `JvmType` and `MethodSignature`
7. implement `TypeResolver`
8. implement `TypeRelations`, starting source-first if needed but finishing with external hierarchy support
9. update Java indexing with typed metadata and modifiers
10. update Groovy indexing with typed metadata and `DynamicType`
11. migrate dotted completion to receiver-type-aware lookup
12. add overload and constructor selection
13. expand integration coverage

## Main Risks

- overbuilding a compiler-grade type system before the completion/definition paths actually use it
- collapsing Groovy unknowns into `Object` and making dynamic completion worse
- duplicating type logic in Java, Groovy, and classpath code instead of keeping it in `jvmpls-core`
- leaving member lookup implicit and reintroducing string parsing in completion paths
- changing method FQN semantics mid-phase and breaking existing index lookups
- trying to solve every generic edge case before landing the common cases
- removing string signatures too early and breaking current query paths

## Success Criteria

Phase 4 is done when:

- `jvmpls-core` has a shared structured type model
- Java and Groovy source symbols expose typed metadata
- binary JDK/JAR symbols expose typed metadata
- dotted completion resolves receiver members through structured types
- constructors and overloads are ranked by typed compatibility
- existing completion and definition behavior remains intact or improves
