# jvm-pls v1.0.0 Roadmap

## Scope

First release targeting fully working **Java + Groovy** support. Kotlin, Scala, Clojure, and other JVM languages are deferred to future releases.

## Key Decisions

- **LSP transport:** LSP4J 1.0.0
- **Approach:** Vertical slices — end-to-end working server early, deepened iteratively
- **Groovy dynamics:** Compile-time AST transform support + runtime metaprogramming heuristics. Architecture supports framework-specific plugins (Grails, Spock, Gradle DSL) in future releases.
- **Build systems:** Maven (via Alipsa maven-utils library) + Gradle (Tooling API). Pluggable `BuildToolPlugin` interface so Ant, Uso, and others can be added later.
- **Classpath scanning:** ClassGraph for fast class discovery, ASM for detailed type/generic signature reading on demand.
- **Distribution:** Standard JAR (dependencies from Maven Central) + fat JAR (maven-assembly-plugin). No editor extensions — standalone server with `--stdio` transport.
- **First clients:** Gade (Groovy analytic IDE), then Claude plugin.

## LSP Feature Tiers

**Tier 1 — Must ship:**
- `textDocument/completion`
- `textDocument/definition`
- Diagnostics via `textDocument/publishDiagnostics` (parse + semantic)
- `textDocument/hover` (type info + Javadoc/Groovydoc)
- `textDocument/didOpen`, `didChange`, `didClose`
- Workspace initialization with classpath resolution

**Tier 2 — Should ship:**
- `textDocument/references`
- `textDocument/documentSymbol`
- `workspace/symbol`
- `textDocument/signatureHelp`
- `textDocument/codeAction` (auto-import, organize imports, add missing stubs)

**Tier 3 — Additional enhancements:**
- `textDocument/rename` (with cross-language rename)
- `textDocument/formatting` (delegate to external formatters)
- `textDocument/implementation` (find implementing classes)
- `textDocument/typeDefinition`
- Call hierarchy

## Phase 1: LSP Transport & Server Shell

**New module: `jvmpls-server`** — depends on `jvmpls-core`, `jvmpls-java`, `jvmpls-groovy`, and LSP4J 1.0.0.

Phase 1 builds on the existing transport-agnostic `CoreFacade`/`CoreServer` APIs rather than replacing them. `jvmpls-server` is the LSP transport adapter and CLI entry point on top of the current in-process core.

### Deliverables

- Implement LSP4J's `LanguageServer` and `TextDocumentService` interfaces.
- Bridge LSP lifecycle (`initialize`, `shutdown`, `exit`) to `CoreEngine`.
- Map `didOpen`/`didChange`/`didClose` to existing `CoreFacade` methods.
- Wire existing `completions()` and `definition()` through to LSP responses.
- Publish diagnostics returned by `index()` via LSP `publishDiagnostics`.
- Launch via stdin/stdout JSON-RPC (standard LSP transport).
- `Main` class with CLI argument parsing for the fat JAR entry point.
- Keep the current in-process `CoreServer` available for tests and embedding.

### Build & packaging

- `maven-assembly-plugin` in `jvmpls-server` produces the fat JAR (`jar-with-dependencies`).
- Standard JAR published normally via Maven.
- ServiceLoader files merged properly in the fat JAR.

### End state

Launch `java -jar jvmpls-server.jar`, connect any LSP client, open a Java or Groovy file, get completions + definitions + parse diagnostics.

## Phase 2: Classpath & External Type Resolution

**New module: `jvmpls-classpath`** — depends on `jvmpls-core`, ClassGraph, and ASM.

### Components

- **`ClasspathScanner`** — Uses ClassGraph to enumerate all classes on a given classpath (JAR paths + directories). Produces a lightweight index: FQN to JAR location + basic info (kind, superclass, interfaces, modifiers).

- **`BinaryTypeReader`** — Uses ASM to read detailed type info on demand from `.class` files: full generic signatures, method signatures with parameter names, field types, annotation values. Lazy — only reads details when a type is actually queried.

- **`ClasspathSymbolProvider`** — Adapts scanned/read binary types into `SymbolInfo` objects and registers them with `SymbolIndex`. Integrates with the existing `CoreQuery` interface so plugins resolve external types the same way they resolve source types.

- **`JdkIndex`** — Scans the target JDK's `jrt:/` filesystem (Java 9+ module system) to index `java.*`, `javax.*`, etc. Prefer the JDK resolved from the build model/toolchain. Fall back to the running JDK only when no workspace JDK is configured.

### Changes to core

- `SymbolIndex` gets a `registerProvider(SymbolProvider)` mechanism for lazy resolution of classpath symbols.
- `CoreEngine` accepts a classpath + target-JDK configuration and initializes the scanner during workspace startup.

### End state

Completions and definitions resolve JDK types (`List`, `Map`, `String`, etc.) and any JAR on the configured classpath. Package/type completions, imports, hovers, and definitions work for external symbols. Receiver-type-aware dotted completions are completed in Phase 4.

## Phase 3: Build System Plugin Architecture

**New module: `jvmpls-build`** — build tool plugin API.
**New module: `jvmpls-build-maven`** — Maven implementation.
**New module: `jvmpls-build-gradle`** — Gradle implementation.

### Build tool plugin interface

```java
public interface BuildToolPlugin {
    String id();                          // "maven", "gradle", "uso"
    boolean applies(Path projectRoot);    // e.g., pom.xml exists?
    BuildModel resolve(Path projectRoot); // extract the full project model
    default void watch(Path projectRoot, Consumer<BuildEvent> listener) {}
}
```

### BuildModel

What a build tool provides:
- Source roots (main + test, per module)
- Resolved classpath (list of JAR paths)
- Output directories
- Module structure (for multi-module projects)
- JDK version / compiler settings
- Submodule relationships

### Implementations

- **`MavenBuildPlugin`** — Uses Alipsa maven-utils library to programmatically resolve the POM model, dependency classpath, source roots, and module structure. No external `mvn` invocation. Enhancements to maven-utils contributed upstream as needed.

- **`GradleBuildPlugin`** — Uses Gradle Tooling API to connect to the project's Gradle daemon and query the project model (source sets, configurations, resolved dependencies). Handles multi-project builds.

### Discovery

ServiceLoader-based, same pattern as language plugins. `BuildToolRegistry` finds all `BuildToolPlugin` implementations, calls `applies()` on the project root, and chooses via deterministic priority rules (or explicit user configuration). Do not rely on ServiceLoader iteration order.

### Integration

- Resolved classpath from `BuildModel` feeds into `ClasspathScanner` (Phase 2).
- Build file changes trigger classpath re-resolution. Primary mechanism: `BuildToolPlugin.watch()` registers a listener. Fallback: `jvmpls-server` watches all dependency-affecting build files it can discover, including root and module `pom.xml`, `build.gradle`, `build.gradle.kts`, `settings.gradle`, `settings.gradle.kts`, and relevant toolchain/property files, then calls `resolve()` again.
- `jvmpls-server` `initialize` handler runs build tool discovery on `rootUri`.
- Workspace configuration supports manual classpath override as fallback.

### End state

Open a Maven or Gradle project, the server auto-detects the build system, resolves all dependencies, and external types are immediately available.

## Phase 4: Type System & Enhanced Symbol Resolution

**New package: `se.alipsa.jvmpls.core.types`**

### Core type model

- **`JvmType`** (sealed interface) with implementations:
  - `PrimitiveType` — int, boolean, char, etc.
  - `ClassType` — reference by FQN with type arguments
  - `ArrayType` — wraps another `JvmType`
  - `TypeVariable` — generic parameter (`T`, `E`)
  - `WildcardType` — `? extends Foo`, `? super Bar`
  - `VoidType` — for method return types
  - `DynamicType` — Groovy-specific "we don't know" marker. Distinct from `Object`. Signals the completion engine to offer everything plausible rather than nothing.

- **`MethodSignature`** — parameter types, return type, type parameters, throws types, modifiers. Replaces string-based signatures on `SymbolInfo`.

- **`TypeResolver`** — resolves type references in source context to `JvmType`. Handles simple names via imports, FQNs, generic instantiation, arrays, primitives.

- **`TypeRelations`** — subtyping queries: `isAssignableTo`, `commonSupertype`, `isSubtypeOf`. Uses class hierarchy from `SymbolIndex` + classpath data.

### Changes to existing code

- `SymbolInfo` gains `JvmType resolvedType()` alongside existing string signature (backward compatible during transition).
- `CompletionItem` gains type information for client display.
- Both plugins updated to populate `JvmType` when indexing methods/fields.
- `BinaryTypeReader` (ASM) maps class file generic signatures to `JvmType`.

### Method resolution

- Dotted completions (`foo.bar`) look up the type of `foo`, find members of that type, offer them.
- Overload resolution uses parameter types to rank candidates.
- Constructor resolution for `new Foo()` expressions.

### End state

The server understands types as structured objects, navigates type hierarchies, resolves generic instantiations, and provides type-aware dotted completions. Groovy dynamic contexts use `DynamicType` for best-effort behavior.

## Phase 5: Groovy Dynamic Feature Support

### 5A: Compile-Time AST Transformations

**New package: `se.alipsa.jvmpls.groovy.transforms`**

- **`TransformAnalyzer`** interface — given a class AST node and its annotations, produces synthetic `SymbolInfo` entries that the transform would generate.

- **Built-in analyzers:**
  - `@Builder` — synthetic builder class + `builder()` method + setter methods
  - `@Canonical` / `@TupleConstructor` — synthetic constructors
  - `@Delegate` — delegated methods from the delegate type
  - `@Immutable` — constructor, getters, `toString`, `equals`, `hashCode`
  - `@Sortable` — `compareTo` + `comparatorByX()` methods
  - `@AutoClone`, `@AutoExternalize` — `clone()`, `readExternal`, `writeExternal`
  - `@Memoized` — suppresses "unused" warnings
  - `@Slf4j` / `@Log` / `@Log4j` / `@Log4j2` / `@Commons` — synthetic `log` field
  - `@CompileStatic`, `@TypeChecked` — hint to enforce static typing

- **`TransformRegistry`** — maps annotation FQNs to `TransformAnalyzer`. Extensible for framework-specific transforms later.

- **Integration:** During `GroovyPlugin.index()`, after AST walking, transform analyzers run on annotated classes and report synthetic symbols via `SymbolReporter`.

### 5B: Runtime Metaprogramming Heuristics

**New package: `se.alipsa.jvmpls.groovy.dynamic`**

- **`MetaClassAnalyzer`** — recognizes `Foo.metaClass.bar = { ... }` patterns, infers methods/properties added to types.

- **`MethodMissingAnalyzer`** — when a class implements `methodMissing`, marks it as dynamically extensible. Completions fall back to `DynamicType` behavior.

- **`CategoryAnalyzer`** — recognizes `use(FooCategory) { ... }` blocks, makes category methods available on target types within scope.

- **`MixinAnalyzer`** — detects `@Mixin(Bar)` or `Foo.mixin(Bar)`, adds `Bar`'s methods to `Foo`.

- **`PropertyMissingAnalyzer`** — like `MethodMissingAnalyzer` but for dynamic properties.

- **Confidence scoring:** Each heuristic-inferred symbol gets a confidence level (HIGH for metaClass assignments with clear types, LOW for methodMissing). Completions rank heuristic results below statically-known symbols. Diagnostics soften ("possibly undefined" vs "undefined").

### End state

`@Builder` classes show builder methods in completions. `Foo.metaClass.bar = { ... }` makes `bar` appear on `Foo` instances. `methodMissing` classes don't produce false "unknown method" errors. Categories add methods within their `use` block scope.

## Phase 6: Tier 1 & Tier 2 LSP Features

### Tier 1: Must ship

- **`textDocument/hover`** — resolved type, method signature, documentation. Javadoc from Java source, Groovydoc from Groovy source, binary docs from attached source/Javadoc artifacts when available; otherwise show signature/type information only. For `DynamicType` symbols, show inference source (e.g., "inferred from metaClass assignment at Foo.groovy:42").

- **Semantic diagnostics** beyond parse errors:
  - Unresolved symbols (hard error for `@CompileStatic`, soft warning for dynamic Groovy)
  - Type mismatches on assignments/arguments (when types are known)
  - Unused imports
  - Duplicate declarations
  - Access modifier violations
  - Missing/excess method arguments

### Tier 2: Should ship

- **`textDocument/references`** — reverse index: for each symbol FQN, track all reference locations. Built incrementally during `index()` via new `SymbolReporter.reportReference(String targetFqn, Location useSite)`.

- **`textDocument/documentSymbol`** — AST outline (classes, methods, fields) for a single file. Both plugins already walk the AST.

- **`workspace/symbol`** — fuzzy search over `SymbolIndex` by name. ClassGraph index provides external symbols. Prefix/substring matcher over FQN keyset.

- **`textDocument/signatureHelp`** — inside `(` of a method call, show parameter names and types. Uses overload resolution from Phase 4.

- **`textDocument/codeAction`** — quick fixes:
  - Auto-import (already implemented as `TextEdit`, needs LSP wiring)
  - Organize imports (sort + remove unused)
  - Add missing method stub (extending abstract class)

### Changes to core

- `SymbolIndex` gets a reverse reference map (`FQN -> Set<Location>`).
- `SymbolReporter` gains `reportReference()`.
- Both plugins updated to report references during indexing.

### End state

Full Tier 1 + Tier 2 LSP features working across Java and Groovy with cross-language support.

## Phase 7: Tier 3 LSP Features

### Deliverables

- **`textDocument/rename`** — Find all references to a symbol (using the reverse index from Phase 6), compute text edits across all affected files, validate that the rename doesn't introduce conflicts. Cross-language rename: renaming a Java class updates Groovy files that reference it and vice versa.

- **`textDocument/formatting`** — Delegate to external formatters rather than implementing formatting logic. Java: invoke google-java-format or Eclipse formatter. Groovy: invoke Spotless (for example via the Greclipse formatter) or another true formatter. Keep lint/static-analysis tools such as CodeNarc separate from formatting. Configuration via workspace settings for formatter choice and style.

- **`textDocument/implementation`** — Given an interface or abstract class, find all implementing/extending classes. Uses the type hierarchy from `TypeRelations` (Phase 4) and the classpath class index.

- **`textDocument/typeDefinition`** — Navigate from a variable/expression to the definition of its type (as opposed to `definition` which goes to the declaration of the variable itself).

- **Call hierarchy** (`callHierarchy/incomingCalls`, `callHierarchy/outgoingCalls`) — For a method, find all methods that call it (incoming) and all methods it calls (outgoing). Requires tracking call-site references during indexing.

### End state

Full LSP feature coverage across all three tiers for Java and Groovy.

## Phase 8: Incremental Analysis & Performance

### Incremental reanalysis

- **`DependencyGraph` activation** — Both plugins report file-level dependencies during `index()`. When a file changes, the graph identifies affected dependents and triggers reindexing only for those files.

- **Dirty file propagation** — When `Bar.java` changes, re-index direct dependents. Full transitive re-index only when a public API signature changes.

- **Partial document sync** — Support LSP `TextDocumentSyncKind.Incremental`. Apply incremental text edits to `DocumentStore` without full content replacement.

### Caching

- **AST cache** — Both plugins cache their most recent AST per file. On re-index, if only a method body changed, reuse top-level structure. Fall back to full reparse for import/class-level changes.

- **Classpath symbol cache** — Persist ClassGraph scan results + ASM type details to `.jvmpls/` directory. On restart, load from cache, verify against JAR timestamps. Avoids rescanning JARs on every startup.

- **Symbol index snapshots** — Periodic snapshots of in-memory `SymbolIndex` to disk for fast warm restart.

### Concurrency

- Ensure all indexing and classpath operations use the existing virtual thread `Executor`. File indexing parallelized across plugins, classpath scanning on background thread, LSP request handling non-blocking.
- Honor LSP `$/cancelRequest` for long-running operations.

### Benchmarking

- Performance test suite with synthetic projects (100, 1000, 10000 files) measuring:
  - Cold start time (full index)
  - Warm start time (from cache)
  - Single file change propagation time
  - Completion response latency
  - Definition lookup latency

### End state

The server handles projects with thousands of files without noticeable lag. Fast restart via disk caching. Single-file edits don't trigger full-project reanalysis.

## Phase 9: Packaging & Distribution

### Final module structure

```
jvmpls-core            (core API, types, symbol index)
jvmpls-classpath       (ClassGraph + ASM scanning)
jvmpls-build           (build tool plugin API)
jvmpls-build-maven     (Maven impl using Alipsa maven-utils)
jvmpls-build-gradle    (Gradle impl using Tooling API)
jvmpls-java            (Java language plugin)
jvmpls-groovy          (Groovy language plugin)
jvmpls-server          (LSP4J transport, Main class)
jvmpls-it              (integration tests)
```

### Artifacts

- **Standard JAR** (`jvmpls-server-1.0.0.jar`) — published to Maven Central with transitive dependencies. For embedding (Gade, programmatic use).
- **Fat JAR** (`jvmpls-server-1.0.0-standalone.jar`) — produced by `maven-assembly-plugin`. Single launchable file. For standalone use.

### CLI

`Main` class arguments:
- `--stdio` (default) — LSP over stdin/stdout
- `--version` — print version and exit
- `--help` — usage info

### Release checklist

- All unit tests pass
- All integration tests pass (cross-language scenarios)
- Performance benchmarks within acceptable thresholds
- Maven Central publication
- GitHub release with fat JAR attached
- CHANGELOG.md for the release
