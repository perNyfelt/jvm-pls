# Phase 3 Implementation Plan

## Goal

Implement Phase 3 from [`req/v1.0.0-roadmap.md`](/home/per/project/jvm-pls/req/v1.0.0-roadmap.md): build-system plugin architecture and workspace initialization with real classpath resolution.

The target outcome is that `jvmpls-server` starts from `rootUri`, detects Maven or Gradle deterministically, resolves the workspace model, and feeds the resulting classpath/JDK/module structure into the Phase 2 external symbol path. After that, Java and Groovy external symbols should come from the workspace build, not the server process.

## Scope

Phase 3 should cover:

- a new `jvmpls-build` API module
- a new `jvmpls-build-maven` implementation
- a new `jvmpls-build-gradle` implementation
- build tool discovery and selection
- workspace startup integration in the server
- classpath/JDK refresh when build files change
- manual classpath override as fallback

Phase 3 should not try to finish:

- hover, references, workspace symbol, or other later LSP features
- Phase 4 structured type modeling
- full file-watching sophistication beyond dependency-affecting build files

Known limitation for Phase 3:

- classpath scoping is workspace-global rather than per-module, so multi-module projects may expose broader dependency visibility than ideal

## Architectural Direction

Keep the separation introduced in Phase 2:

- `jvmpls-core` stays transport-agnostic
- build tools resolve workspace state into a `BuildModel`
- `jvmpls-server` owns workspace detection, lifecycle, and refresh
- `jvmpls-classpath` continues to consume explicit classpath/JDK input, not discover builds itself

That means Phase 3 should add a build-resolution layer between LSP workspace initialization and the existing `CoreServer`/provider setup, rather than pushing Maven or Gradle logic into `jvmpls-core`.

Two design constraints need to be explicit up front:

- `CoreServer` is effectively immutable after construction today, so bootstrap must be refactored before workspace-backed resolution can be integrated cleanly
- LSP service objects must exist before `initialize()`, but the real workspace-backed core cannot be created until `rootUri` and initialization options are known

## Workstreams

### 1. Add `jvmpls-build`

Create a new API module:

- `jvmpls-build`

Primary types:

- `BuildToolPlugin`
- `BuildModel`
- `BuildModule`
- `BuildEvent`
- `BuildToolRegistry`

`BuildModel` should stay pragmatic and include:

- project root
- source roots
- test source roots
- classpath entries
- output directories
- module list
- target JDK home/version when available
- dependency-affecting files to watch

### 2. Define deterministic build tool selection

`BuildToolRegistry` should use `ServiceLoader`, but selection must be explicit and stable.

Suggested rules:

- explicit user override wins
- otherwise prefer the most specific matching tool at the workspace root
- if both apply, fail with a logged warning unless rules clearly resolve it

Do not rely on `ServiceLoader` iteration order.

### 3. Implement `jvmpls-build-maven`

Use the Alipsa `maven-utils` path described in the roadmap.

Resolve:

- reactor/modules
- main/test source roots
- compile/test classpaths
- target output directories
- toolchain/JDK info when available

Avoid shelling out to `mvn`.

### 4. Implement `jvmpls-build-gradle`

Use the Gradle Tooling API.

Resolve:

- projects/subprojects
- source sets
- resolved compile classpath
- output directories
- JVM/toolchain info

Support both `build.gradle` and `build.gradle.kts`.

### 5. Add server-side workspace bootstrap

Replace the current Phase 3 placeholder in [`jvmpls-server/src/main/java/se/alipsa/jvmpls/server/JvmPlsLanguageServer.java`](/home/per/project/jvm-pls/jvmpls-server/src/main/java/se/alipsa/jvmpls/server/JvmPlsLanguageServer.java) with:

- discover build tool from `InitializeParams.rootUri`
- resolve `BuildModel`
- build a `CoreServer` using the resolved classpath and target JDK
- log selected tool, root, classpath size, and fallback paths

This is not a small follow-on refactor. It requires an explicit split between:

- long-lived `TextDocumentService` and `WorkspaceService` shells created at server construction time
- a swappable core delegate installed during `initialize()`

Recommended behavior before initialization completes:

- document and workspace services exist and are LSP-safe
- requests that need the core are rejected cleanly until the core delegate is installed
- buffering should be avoided unless there is a concrete need

### 6. Add workspace re-resolution

Start with the LSP client as the primary trigger.

Recommended Phase 3 mechanism:

- use `workspace/didChangeWatchedFiles` notifications from the client for dependency-affecting files

Optional later enhancement:

- build plugin may expose `watch(...)`
- server-side `WatchService` fallback for discovered files:
  - `pom.xml`
  - `build.gradle`
  - `build.gradle.kts`
  - `settings.gradle`
  - `settings.gradle.kts`
  - toolchain/property files the plugin reports

On change:

- re-run `resolve()`
- rebuild external providers/classpath config
- re-analyze open documents

### 7. Add manual override path

Support explicit configuration when build resolution fails or is unsupported.

Likely shape:

- LSP initialization options or server config object with:
  - `classpath`
  - `targetJdkHome`
  - optional `buildTool`

Manual config should bypass auto-detection cleanly.

Precedence rules should be explicit:

- manual override wins over auto-detected build tool output
- explicit build tool selection wins over registry auto-selection
- if neither applies, fall back to JDK-only resolution

### 8. Refactor core bootstrap for reconfiguration

Phase 2 wired external providers at `CoreServer` creation time.

This is a prerequisite for server integration, not a cleanup task to do afterwards.

Phase 3 needs a reusable bootstrap/factory object so the server can swap workspace classpath/JDK without entangling the transport layer.

A good direction is a `CoreServerFactory` or `WorkspaceCoreFactory` in `jvmpls-core` or `jvmpls-server` that takes explicit:

- classpath
- target JDK
- diagnostics publisher

The main requirement is that the server can replace the workspace-backed core when:

- initialization first resolves the workspace
- build files change and classpath/JDK need re-resolution
- manual configuration is updated

### 9. Define resolution failure behavior

Build resolution can fail for normal user reasons:

- corrupt or incomplete POMs
- missing Maven parent/BOM metadata
- Gradle Tooling API connection failures
- wrapper or daemon problems

Phase 3 should degrade gracefully:

- LSP initialization still succeeds
- the server logs a clear warning
- the workspace runs with JDK-only symbols unless manual overrides are supplied
- diagnostics/completions/definitions continue to work for open source and JDK types

Do not fail server startup just because build resolution failed.

### 10. Define Gradle runtime assumptions

Phase 3 should not bundle a Gradle distribution into `jvmpls-server`.

Recommended assumptions:

- prefer the project Gradle wrapper when present
- otherwise allow a locally installed Gradle reachable by the Tooling API
- if neither is available, Gradle resolution fails gracefully and the server falls back as described above

This keeps packaging and the shaded JAR size under control.

### 11. Test strategy

#### Unit tests

Add focused tests for:

- build tool selection rules
- `BuildModel` mapping
- fallback/manual override precedence
- graceful degradation on resolution failure

#### Maven tests

Add tests for:

- single-module project
- multi-module reactor

#### Gradle tests

Add tests for:

- single-project build
- multi-project build

#### Server integration tests

Add end-to-end tests proving:

- `initialize(rootUri=...)` resolves workspace classpath
- external symbols come from workspace dependencies
- `workspace/didChangeWatchedFiles` triggers re-resolution
- fallback manual classpath works when no build tool applies
- initialization succeeds even when Maven or Gradle resolution fails

## Recommended Sequence

Implement in this order:

1. Add `jvmpls-build`
2. Add `BuildToolRegistry` and deterministic selection
3. Refactor bootstrap so the server owns long-lived LSP services but can install or swap a core delegate after `initialize()`
4. Define resolution failure behavior and manual override precedence
5. Implement `jvmpls-build-maven`
6. Implement `jvmpls-build-gradle`
7. Wire `initialize()` to resolve and create workspace-backed core
8. Add `workspace/didChangeWatchedFiles` refresh for dependency-affecting files
9. Add end-to-end tests

## Risks

### Pushing build-tool logic into `jvmpls-core`

Build discovery belongs at the workspace/server layer. If Maven and Gradle logic leak into `jvmpls-core`, the transport-agnostic design gets muddied quickly.

### Overcomplicating refresh

Classpath refresh should be correct and explainable before it is clever. Start with explicit re-resolution from LSP watched-file notifications rather than building an opaque server-side sync system.

### Wrong tool selection in mixed repos

Repos can contain both `pom.xml` and Gradle files. Selection rules need to be deterministic, logged, and overridable.

### Workspace-global classpath visibility

The first Phase 3 implementation will likely apply one resolved classpath to the whole workspace. That is acceptable for the first cut, but it should be documented and tested as a known limitation rather than mistaken for proper per-module scoping.

### Gradle environment assumptions

Gradle resolution depends on wrapper or local tooling availability. The server must not assume it can download or embed a distribution implicitly.

### Scope creep into later phases

Phase 3 should deliver real workspace-backed classpaths. It does not need to solve hover, references, full semantic workspace modeling, or structured type systems yet.
